# PRD — 越南家人「語音查火車 + 相近站名用選的」

## 背景
越南家人（陳氏梅）主要用**語音**、越南語跟 bot 互動。台鐵站名的越南語漢越音 / 口語拼寫
常常對不齊（尤其語音辨識後更容易走音），目前查不到就只回「站名無法辨識」，
使用者不知道要改成什麼。希望：**查不到時，後台 AI 自動列出幾個「最相近的真實車站」，
做成可以「按一下就選」的選項**，越南家人點一下就查到班次，不必再打字。

## 使用者故事
1. 陳氏梅用越南語**語音**說「ngày mai đi tàu từ ga Trúc... đến Trung Lịch」→ bot 聽打成文字後，
   若「Trúc…」無法唯一對應到某站，bot 用越南語回「您是指哪一站？」並列出
   〔新竹〕〔竹南〕〔竹北〕等**可點選按鈕**；她點〔新竹〕→ bot 直接回覆「明天 新竹→中壢」的班次。
2. 打字也一樣：站名打不精準時給選項；點或再打一次都能完成查詢。

## 範圍（要做）
- **F1 語音查火車**：確認「語音 → Whisper 轉文字 → 既有 AI/工具流程 → 台鐵查詢」端到端可用
  （目前理論上已通；本案要確保「語音走音的站名」也能進入 F2 的選站流程，且語音回覆若帶選項，
  選項按鈕不會被語音前綴包裝弄丟）。
- **F2 相近站名選擇（本案重點）**：
  - 當台鐵查詢中某一站**無法唯一辨識**時，後台用模糊比對從全台鐵 245 站找出**最多 5 個最相近**的
    真實車站，作為 **LINE Quick Reply 可點選按鈕**（在使用者語言下）回覆，請使用者選擇。
  - 使用者**點選**（或改打站名 / 回覆數字）後，沿用原本查詢的「另一站 + 今天/明天」完成查詢並回班次。
  - 比對需涵蓋：中文站名、英文站名、越南語漢越音別名（皆去聲調 / 小寫 / 去空白後比對）。
  - 若能高信心唯一命中 → 照舊直接查（不要沒事就跳選單）；完全找不到相近 → 回既有「請提供明確站名」訊息。
  - 主要服務 **AI / 語音 / 自然語言路徑**（越南家人用這條）；中文關鍵字指令路徑（「台鐵 X Y」）
    至少不可回歸，若能一併支援選單更好（次要）。

## 不做（範圍外）
- 不接需付費金鑰的 API（維持 keyless：站名清單用既有 PTX Station 端點）。
- 不做跨日（今天/明天以外）、不做誤字自動「猜一個直接查」（要讓使用者選，避免猜錯）。
- 選擇狀態用記憶體即可（Render 重啟會清空可接受；設 TTL 例如 3 分鐘、每人一筆）。

## 技術現況（給架構師的已知事實）
- 回覆管線目前只傳「字串」：`handler.replyForEvent(event)` 回字串 →
  `src/index.js` `handleEvent` 包成 `{ type:'text', text }` 用 `lineClient.replyMessage` 送出。
  要支援按鈕，需讓「回覆」能夾帶 quickReply（建議：handler 可回 `string` 或
  `{ text, quickReply }`；index.js 依型別組訊息）。
- **LINE Quick Reply 格式**（@line/bot-sdk v11 messagingApi）：訊息物件加上
  `quickReply: { items: [ { type:'action', action:{ type:'message', label:'新竹', text:'新竹' } }, ... ] }`。
  最多 13 顆、label ≤ 20 字；點下去會把 `text` 當成使用者訊息送回（畫面也顯示該 text）。
- 站名解析在 `src/services/traTrain.js` `normalizeStation`（已支援中/英/越南語別名、async、回 `{name,id}|null`），
  站表由 `fetchStations()`（PTX 245 站、快取 24h）提供。AI 工具走 `getTraTrainSummary`；工具定義在 `src/tools.js`。
- 語音在 `handler.handleAudio`：轉文字後呼叫 `handleText`，再用 `lang.audioPrefix` 前綴包裝回覆字串。
- 對話記憶 `src/conversation.js`（記憶體）。多人共用同一支 bot，`userId` 區分。

## 驗收條件（可測，tester 依此驗收）
1. **模糊比對** `suggestStations(token, max)`（或等效函式）：
   - 給「走音 / 不完整」的輸入能回相近真站清單（每筆含 `name`、`id`），按相近度排序、去重、≤ max：
     - `'tan chu'` / `'truc'` → 候選含 新竹（且排前）
     - `'sin zu'` / `'hsin'` → 候選含 新竹
     - 明顯亂碼（如 `'zzzzzz'`）→ 回空陣列
   - 已能唯一辨識的（`'新竹'`、`'tân trúc'`、`'Hsinchu'`）→ 照 `normalizeStation` 直接命中，**不**進選單。
2. **選單流程（狀態機）**：
   - 給一個「一站可唯一辨識、另一站需要選」的查詢，系統會產生 pending 選擇狀態（含已知站、日期、候選清單），
     並回一則帶 `quickReply`（≥2、≤5 顆按鈕）的訊息。
   - 接著送出「點選其中一顆按鈕的站名（模擬 LINE 回傳該 text）」→ 系統用 pending 完成查詢，
     回台鐵班次字串（含兩站中文名），並清掉 pending。
   - pending 過期（超過 TTL）或送出不相干訊息 → 不誤判為選擇，走正常流程。
3. **回覆管線**：`index.js` 能把 `{ text, quickReply }` 正確送成帶 quick reply 的 LINE 訊息；
   一般字串回覆行為不變（回歸測試：隨便一句聊天仍正常）。
4. **語音**：`handleAudio` 對「內容會觸發選站」的語音，回覆需保留 quickReply（不可因語音前綴而遺失按鈕）。
5. **不回歸**：既有「台鐵 台北 台中」「下一班 台北到花蓮」、越南語「tân trúc → trung lì 明天」等仍正常。
6. `node --check` 全部變更檔通過。
