# PRD — 修復越南家人查台鐵時刻「站名辨識失敗」bug

## 背景 / 真實回報
越南家人在 LINE 問「Ngày mai tân trúc đến trung ly có chuyến xe mấy giờ」（明天 新竹→中壢 有幾點的車），
bot 卻回中文「無法找到從『和林』到『中壢』的火車班次」，甚至第二次變成「花蓮→中壢」。
連問三次都失敗，最後回「抱歉，我現在無法回覆」。

## 根因（已診斷）
1. **主因：AI 被指示「傳中文站名」。** `src/tools.js` 的 `timeContext()` 尾句寫
   「呼叫 get_tra_train 工具（傳中文站名）」，且 `from/to` 參數說明「優先用中文站名」。
   於是模型把越南語站名**自己音譯猜成漢字**——「Tân Trúc」猜成「和林 / 花蓮」（錯，應是新竹）——
   再把猜錯的漢字丟給工具。伺服器端的 `VN_ALIAS`（`'tan truc'→'新竹'`）根本沒機會作用，
   因為它只收到猜錯的中文。
2. **次因：口語拼寫變體。** 使用者打「trung lì」→ 去聲調後為 `trung li`，
   但 `VN_ALIAS` 只有 `trung lich` / `trung ly`，少了 `trung li`，仍可能漏接。

## 目標
越南語（含隨手拼寫、缺聲調）站名能被正確辨識並查到班次；模型不再自行音譯猜漢字。

## 範圍
- 僅改 `src/tools.js`（工具說明 / timeContext）與 `src/services/traTrain.js`（VN 別名 + 容錯）。
- 不改動查詢邏輯、快取、輸出格式、其他工具。

## 驗收條件（可測）
1. `normalizeStation` 對以下輸入都回正確站名物件（不需上線，離線純函式即可測，
   但需能連 PTX 站表；測試允許 mock 或實連）：
   - `Tân Trúc` / `tân trúc` / `tan truc` → 新竹
   - `Trung Lịch` / `trung lich` / `trung ly` / `trung lì` / `trung li` → 中壢
   - `Đài Bắc` / `dai bac` → 台北；`Cao Hùng` → 高雄
   - 中文原樣 `台北`、英文 `Hsinchu`、簡稱 `北車` 仍照舊可辨識（不回歸）
2. `src/tools.js` 的 `timeContext()` **不再**出現「傳中文站名」字樣；改為指示
   「傳使用者原本說的站名原文（中/英/越南語皆可），不要自己音譯成漢字」。
3. `from`/`to` 參數說明同樣不再要求「優先中文」，改為「用原文、勿自行音譯」。
4. `node --check` 兩檔皆通過；既有中文指令（`台鐵 台北 台中`）行為不變。
5. 端到端：模擬那句越南語 → 工具收到的是原文/正確中文，最終查到「明天 新竹→中壢」班次。
